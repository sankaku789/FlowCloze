# ソフトウェア工学の概論
#qblock{
- [情報システム]{term-name}は，人，機械，コンピュータが互いに協調してある[目的を達成する仕組み全体]{meaning}のこと
}

#qblock{
- [ソフトウェア]{term-name}は[プログラム]{relation}，[データ]{relation}，関連する[ドキュメント]{relation}
}

- 情報処理システムは，情報システムの内，コンピュータから構成される仕組み
- ソフトウェアは，情報システム上で動くのを用意することが目的

## 情報システムと組込みシステム
#qblock{
- [情報システム]{term-name}は，人間の[業務を支える]{meaning}もの
  * 相手は非専門家であることが多い．
  * [使いやすさ]{meaning}や[わかりやすさ]{meaning}が重要
- [組込みシステム]{term-name}は，[物理現象に働きかける]{meaning}もの
  * 相手も専門家であることが多い
  * [安全性]{meaning}や[リアルタイム性]{meaning}が重要
}

## ソフトウェアの特徴
#qblock{
- ソフトウェアは対象次第での作り変えなどにより，[多種多様]{meaning}になる
- 要求が増加し，様々な機能が追加されていき，[大規模化]{meaning}，[複雑化]{meaning}している．
- ビジネスモデルの変化や，技術の更新に適応しなければならず，[変化が激しい]{meaning}
}

## ソフトウェアの良し悪し
#qblock{
- [QCD]{term-name}は[品質]{relation}，[コスト]{relation}，[納期]{relation}


- ソフトウェアの良し悪しは = プロジェクトの成否
- 規模の増大により計画どおりに進みづらくなる
}

## ソフトウェア工学とは
#qblock{
- [ソフトウェア工学]{term-name}は，コストなどの制約のもと，開発利用するための[理論等を整理したもの]{meaning}
  * ソフトウェア開発に関わる問題を[解決・改善]{meaning}することが目的
- 経験則に基づいている
- コンピュータの性能向上により，情報システムが構築されはじめた
- 結果，開発のQCDなどが問題になる
- [ブルックスの法則]{term-name}「遅れている開発プロジェクトでは，開発要員の増員は遅れをひどくする」
- [工数]{term-name} = [人数]{relation} × [期間]{relation}
- 現代では，性能やメモリ利用効率を落としても，読みやすく修正しやすいコードが必要
- ソフトウェアモデルは，プログラミング言語より抽象的に作成
- 開発方法(開発の際の枠組み)や開発環境(IDEやコミュニケーションツール)も需要となる
- コードやドキュメントの再利用やパターンを使うことで資産を生かす(ライブラリなど)
}

## 工程
#qblock{
- [上流工程]{term-name}は，[要求定義]{process}と[設計]{process}
   * 要求定義が，顧客から要求を聞き，要求事項としてまとめる
   * 設計はコードを記述するための方法を定義
- [下流工程]{term-name}は，[実装]{process}と[テスト]{process}
}

## 大規模化に伴う問題
### 上流工程の問題
#qblock{
- 不具合の多くは[下流工程]{process}にて生まれる．
- しかし，原因は[上流工程]{process}にて生まれる
- この場合，上流工程からやり直す[rework]{term-name}が発生し，工期に影響する

### 管理や組織の問題
- [プロジェクト]{term-name}は，時間，コスト，資源が限られている活動
- [プロジェクト管理]{term-name}は[QCD]{relation}の管理
- 規模の増大により，調整を行う管理業務の重要性が増す
- コミュニケーション力は重要なスキルとなる

### 変化の問題
- ソフトウェアは再利用されることが多い
- 再利用のために，ソースを追加，編集していく
  * 結果，読みづらくなる
- [レガシーコード]{term-name}は，業務の中核を支える
- 長期間の修正の繰り返しで品質が劣化
  * これをソフトウェアは「[慣性]{meaning}」を持つ
}


# 情報システムとソフトウェア
## 情報とデータ
#qblock{
- [データ]{term-name}は，事実や概念，指示の[表現]{meaning}
  * ex) 文字コード
- [情報]{term-name}は，データを評価して得られる[意味]{meaning}
}
- Infomation = function(human)

## ソフトウェア動作の特性
#qblock{
- ソフトウェアの動作は[離散的]{meaning}
  * CPUはCLKで飛び飛びの値しか表現できない
- [インタリーブ]{term-name}(一時停止)して動作
  * OSがプログラムを切り替えながら動作
- 最終的に機械語で実行
  * インタリーブも機械語単位
}

## 情報システムの構成
#qblock{
- [クライアントサーバ方式]{term-name}
  * クライアント -> サーバ
  * クライアントの方が多い
- [集中処理]{term-name}は，1台のサーバで処理
  * 効率的でデータの一貫性を保つことができる
  * しかし，故障に弱く，1台に負荷が集中
- [分散処理]{term-name}は，複数のサーバに処理
  * 処理の分担により，負荷の軽減ができる他，バックアップ
  * データの一貫性やコストの問題e
}

### 可用性向上のためシステム構成
#qblock{
- [シンプレックスシステム]{term-name}は，1台のコンピュータでシステムを構築
  * 低コストだが，故障するとDead
- [デュプレックスシステム]{term-name}は，[主系]{relation}と[従系]{relation}でシステムを構築
  * 通常は別々の処理を実行，故障時は従系を使う
  * 高コストだが，故障にはつよい
- [デュアルシステム]{term-name}は，2台のコンピュータでシステムを構築
  * どちらも同じ処理を行う
  * 高コストで効率は悪い
}

## 組込みソフトウェアの特徴
#qblock{
- 専用ハードウェアであるため，ハードウェアへの依存が強く，[移植性]{meaning}が悪い．
- イベントが発生してから，それに応じて応答する([リアクティブ性]{meaning})
- 時間制約を守る必要([リアルタイム性]{meaning})
- [資源の制約]{meaning}
- 継続して機能し続ける性質([信頼性]{meaning})
- 人・モノ・環境に損害を与えない性質([安全性]{meaning})
}

# 要求定義
#qblock{
- [要求定義]{term-name}は，「顧客が欲しいモノ」から，「[要求仕様書]{relation}」をまとめる工程
- プロセスは[要求獲得]{process} -> [要求分析]{process} -> [要求仕様化]{process} -> [検証]{process}
- [要求]{term-name}は，問題を解決したり，目標を達成するために顧客が必要とする条件や能力
- 要求は，[機能要求]{term-name}と[非機能要求]{term-name}にわけられる
  * 機能はソフトウェアが提供するサービス
  * 非機能は品質や制約，法令要求
  * 厳密な区別はない
- [ステークホルダ]{term-name}は，システムによって影響を受ける/与える人，グループ，組織のこと
  * ステークホルダごとに要求の重要度は異なる
- 要求仕様書をまとめる．
  * 本当に難しい
}

## 要求獲得
#qblock{
- [要求獲得]{term-name}では顧客から情報収集する
- [インタビュー]{process}
  * 深堀りできるが，情報が未整理
- [アンケート]{process}
  * 発展性もない
- [ワークショップ]{process}
  * 複数のステークホルダで議論して，気づきにくい要求を獲得しうる
- [ブレーンストーミング]{process}や[プロトタイピング]{process}で発想支援
- ステークホルダ間で利害が一致しない場合，「落とし所」をみつける
}

## 要求分析
#qblock{
- [ゴール分析]{process}は，目標を設定し，認識・整理する
  * 直接満たすは難しいのでサブゴールを設定
  * [AND分割]{term-name}は下位がすべて満たされると，上位もみたされる
  * [OR分割]{term-name}は下位のうち1つでも満たされると，上位も満たされる
- [シナリオ分析]{process}は，ゴールよりも具体的に記述する
  * シナリオは，一連の出来事を時系列に記述
  * システム利用の流れや振る舞いの理解に有用
  * シナリオの記述方法
    - [叙述的なシナリオ]{relation}は，日常的な言葉で文章として書く(非専門家との議論で有効)
    - [構造的なシナリオ]{relation}は，箇条書きや表で表現(時間的な関係を明確にできるが，複雑な条件は厳しい)
    - [ソフトウェアモデルの利用]{relation}は，UMLのシーケンサ図やアクティビティ図に沿ってモデルを描く
      * 正確かつ厳密な記述
      * 読み書きに一定のなれ
- [ユースケース分析]{process}
  * システムの利用のされ方を記述
  * システムが提供する機能をグループ化できる
  * シナリオ以上ゴール以下の抽象度
  * システムを粗く把握
- 上記の方法は，どれか一つに絞る必要なし
}

## 要求仕様化
#qblock{
- [要求仕様化]{term-name}は，要求分析した結果を整理・構造化して「ソフトウェアへの要求」にする
- 顧客 -> 技術者に誤解なく伝える必要
- ソフトウェア開発の入力であるため，厳密で一般的な記述が必要
- 満たすべき性質
  * [正当性]{meaning}(満たすべき要求のみ記述)
  * [非あいまい性]{meaning}(要求は一意に解釈)
  * [完全性]{meaning}
  * [無矛盾性]{meaning}
  * [順序付け]{meaning}
  * [検証可能性]{meaning}
  * [変更可能性]{meaning}
  * [追跡可能性]{meaning}(根拠が明確)
}

# 設計
## 設計の基礎
#qblock{
- [設計]{term-name}は，「要求仕様書」から「[設計書]{relation}」を作成する工程
- ソフトウェアの[構造]{relation}，構成要素とその関係や特性を決める
- [外部インターフェース]{relation}の決定
  * 外部設計ともいう
- [内部の実現方法]{relation}の決定
  * 構成要素とその関係をきめる
  * 内部設計(アーキテクチャ設計)ともいう
- [実現技術]{relation}の決定
  * ハードウェアやOS，使用技術をきめる
- 設計の確認
- プログラムの実現は，データ構造とアルゴリズム
}

## 要求と設計の関係
#qblock{
- 要求が[What]{meaning}を定義し，設計は[How]{meaning}を定義する
  * 要求は設計に対応する必要がある
- 品質に影響を与える
- プラットフォームは制約
- 要求に対して，設計を割り当てると新たな問題が発生する可能性
}

## モジュール化
#qblock{
- [モジュール]{term-name}はソフトウェアの構成要素
- [モジュール化]{term-name}とは，モジュールとその関係性を定義
- モジュール化の良し悪しは，品質に直結
- 作業分担の単位にもなる
- 再利用できる構造にする必要 -> [モジュール性]{meaning}がよい
- 適切な規模に分割
}

## 構造化プログラミング
#qblock{
- [構造化プログラミング]{term-name}は，[逐次]{process}，[分岐]{process}，[反復]{process}にて記述
}
- 実行速度を多少おそくしてでも，可読性を重視

## モジュール性
### モジュール強度
#qblock{
- [モジュール強度]{term-name}は，モジュール内の結びつきの強さ
- 強度が強い(役割が強い)ほど，良い
- 以下，種類(強い順)
  * モジュールが1つの機能のみ提供する[機能的]{relation}
  * 特定データへのアクセスを行う機能をまとめた[情報的]{relation}
  * 手順的に共通データを参照・変更する[連絡的]{relation}
  * 順番に実行される複数の機能を手順として1つのモジュールにする[手順的]{relation}
  * ある時点で実行する機能がまとまっている[時間的]{relation}
  * 複数機能で1つが選択可能な[論理的]{relation}
  * 無秩序に機能を寄せ集めた[暗号的]{relation}
}

### モジュール結合度
#qblock{
- [モジュール結合度]{term-name}は，モジュール間の結びつきの強さ
- 結合度が弱い(関係があまりない)ほうが良い
- 以下，種類(強い順)
  * [内容結合]{relation}(他のモジュールないの構成要素に直接アクセス)
  * [共通結合]{relation}(複数のモジュール間で読み書き可能な構造化されたデータ領域を参照)
  * [外部結合]{relation}(複数のモジュール間で読み書き可能な変数を参照している場合)
  * [制御結合]{relation}(他のモジュールの特定の制御を目的として，if 文などの条件式で使用される引数を使う)
  * [スタンプ結合]{relation}(共有変数では構造体データを受け取る場合)
  * [データ結合]{relation}(共有変数でない変数を引数とする場合)
- モジュール間で情報隠蔽が重要(公開された操作のみを許可)
  * これを実現するのが[カプセル化]{term-name}
}

## 機能に注目する設計
#qblock{
- [機能]{relation}(プロセス)は入力データを出力データへ変換
- [機能分割]{process}は，システム全体を1つの機能として，その機能を細分化する
- 最初は抽象的に，段階的に具体的な機能分割をする([段階的詳細化]{process})

## 情報・データに注目する設計
- [情報・データに注目する設計]{relation}は，データ同士の関係を定義
- 詳細化の面では機能分割に近い
- データべ―ス

## 状態に注目する設計
- [状態に注目する設計]{relation}では，イベントと現在の状態で振る舞いと次の状態がきまる(リアクティブシステム)
- [ステートマシン図]{relation}(状態遷移図)は遷移の全体像を直感的に捉えられる
- [状態遷移表]{relation}は考慮漏れがないか確認しやすい
}

## オブジェクト指向設計
#qblock{
- [オブジェクト指向設計]{term-name}は，クラス(設計図)やインスタンスをモジュール化の単位とする(独立性の向上が目的)
- [クラス]{term-name}はメソッドとフィールド(データ)，メソッド(振る舞い)を定義
- [カプセル化]{term-name}に基づき，setterとgetterを介してのみフィールドへのアクセスする
- [継承]{term-name}はスーパークラスの持つフィールドとメソッドをサブクラスにもつ
- [多態性]{term-name}
  * サブクラスで，スーパークラスのメソッドを再定義可能(オーバーライド)
- [余談]渡されるデータの型が異なっても型に応じた処理を実行することもできる([オーバーロード]{term-name})
}
